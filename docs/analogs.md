# Аналоги HiDL: исследование и выводы

Цель: собрать лучшие паттерны мотивации и ответственности, убрать шум, усилить чувство "меня реально сопровождают".

## Подробно по аналогам

### Focusmate
- **Позиционирование:** “Virtual body doubling for getting anything done.”
- **Ключевая механика:** парные сессии, присутствие с камерой, короткий чек‑ин и чек‑аут.
- **Сила:** минимальное трение + ощущение «меня видят» → ответственность без жёсткого контроля.
- **Вывод для HiDL:** режим «Кафе фокуса» = короткая сессия с обязательным чек‑ин/итогом, минимум текста.

### Caveday
- **Позиционирование:** виртуальные coworking‑сессии (The Cave) для фокуса.
- **Ключевая механика:** ведущий, структура сессии (план → спринты → отчёт), групповая динамика.
- **Сила:** ритуал и структура задают темп, удерживают внимание.
- **Вывод:** у HiDL может быть «ведущий‑режим» — бот модерирует сессию и держит тайминг.

### Flow Club
- **Позиционирование:** «Work solo, not alone» — виртуальный coworking и body‑doubling.
- **Ключевая механика:** короткий план, затем сессия, затем отчёт.
- **Сила:** мягкая социальная подотчётность.
- **Вывод:** короткий вопрос «что делаешь?» + «по плану?» лучше длинных чек‑листов.

### Beeminder
- **Позиционирование:** анти‑прокрастинация через деньги и график.
- **Ключевая механика:** контракт, график прогресса и «штраф» при срыве.
- **Сила:** мгновенная мотивация через loss aversion.
- **Вывод:** можно сделать мягкий «контракт» без денег (потеря доступа/привилегий/теплоты).

### StickK
- **Позиционирование:** commitment‑контракты с «рефери».
- **Ключевая механика:** пользователь задаёт цель, ставку и назначает наблюдателя.
- **Сила:** внешний наблюдатель усиливает ответственность.
- **Вывод:** у HiDL может быть «доверенный режим» (друг/второй аккаунт) или «публичный счётчик».

### Habitica
- **Позиционирование:** жизнь как RPG.
- **Ключевая механика:** задачи → опыт/золото; провалы → урон.
- **Сила:** сильная поведенческая петля (reward/punish).
- **Вывод:** короткие игровые последствия (стрик, уровень доверия, «настроение Хидл»).

### Forest
- **Позиционирование:** фокус‑таймер; растёт дерево, если не отвлекаешься.
- **Ключевая механика:** простая визуальная ставка (дерево погибает при срыве).
- **Сила:** простая, мгновенная и понятная цена за отвлечение.
- **Вывод:** «мини‑ставка» на фокус‑сессию работает лучше десятков напоминаний.

### Finch
- **Позиционирование:** self‑care через заботу о питомце.
- **Ключевая механика:** минимальные шаги → развитие питомца.
- **Сила:** эмоциональная привязка и бережный тон.
- **Вывод:** HiDL может быть «суровой», но в моменты провала полезнее бережный тон.

### Todoist
- **Позиционирование:** таск‑менеджер для работы и жизни.
- **Ключевая механика:** быстрый захват задач, проекты, фильтры/виды.
- **Сила:** минимальные действия и ясная структура без шума.
- **Вывод:** нужен сверх‑быстрый ввод и 1‑2 ключевых вида, остальное — в фоне.

### TickTick
- **Позиционирование:** задачи+календарь+привычки в одном приложении.
- **Ключевая механика:** единый список дел, календарь, привычки.
- **Сила:** всё в одном месте, не надо прыгать по разделам.
- **Вывод:** HiDL должен “собирать” задачи/напоминания в 1 ленте и выдавать минимум действий.

### Sunsama
- **Позиционирование:** ежедневный планировщик, который помогает сохранять спокойствие и фокус.
- **Ключевая механика:** daily‑план и time‑boxing.
- **Сила:** пользователь видит реалистичный план дня.
- **Вывод:** HiDL нужен «план‑на‑сегодня» из 1‑3 задач, а не длинные списки.

### Akiflow
- **Позиционирование:** time‑blocking планер.
- **Ключевая механика:** инбокс → блоки в календаре.
- **Сила:** визуальный контроль дня.
- **Вывод:** предлагать 1‑клик «заблокировать слот», не требуя сложного планирования.

### Motion
- **Позиционирование:** AI‑суперапп для задач и календаря.
- **Ключевая механика:** авто‑расписание задач в календаре.
- **Сила:** убирает необходимость вручную планировать время.
- **Вывод:** HiDL должна сама предлагать время, а не спрашивать пользователя.

### Reclaim
- **Позиционирование:** AI‑календарь для работы и жизни.
- **Ключевая механика:** автопланирование задач, привычек и перерывов.
- **Сила:** встраивание задач в реальные окна времени.
- **Вывод:** HiDL может делать «умные окна» (2‑3 предложенных тайм‑слота).

### RescueTime
- **Позиционирование:** автоматический учёт времени + фокус‑сессии.
- **Ключевая механика:** пассивный трекинг, отчёты, фокус‑сессии.
- **Сила:** минимум ручных действий; данные сами показывают правду.
- **Вывод:** меньше ручных чек‑листов, больше автоматических сигналов.

### Freedom
- **Позиционирование:** блокировка сайтов/приложений для фокуса.
- **Ключевая механика:** принудительное отключение отвлечений.
- **Сила:** убирает соблазн, а не убеждает.
- **Вывод:** для «кафе фокуса» нужен “замок”: без завершения сессии — ограничение доступа.

### Cold Turkey
- **Позиционирование:** жёсткий блокировщик отвлечений.
- **Ключевая механика:** не даёт «соскочить» до конца сессии.
- **Сила:** силовой режим помогает прокрастинаторам.
- **Вывод:** у HiDL должен быть “жёсткий режим” для тех, кому нужно.

### Streaks
- **Позиционирование:** простой хабит‑трекер со стриками.
- **Ключевая механика:** серия выполнений.
- **Сила:** простая мотивация, минимум текста.
- **Вывод:** один видимый счётчик прогресса важнее длинных списков.

### Fabulous
- **Позиционирование:** self‑care, рутины и привычки.
- **Ключевая механика:** пошаговые ритуалы и микро‑шаги.
- **Сила:** мягкая мотивация и прогресс через маленькие действия.
- **Вывод:** «микро‑задачи» + поддержка эффективнее, чем “сделай всё”.

### Rize
- **Позиционирование:** автоматический учёт времени (AI).
- **Ключевая механика:** пассивный трекинг.
- **Сила:** честные данные без ручного ввода.
- **Вывод:** чем меньше ручной работы у пользователя, тем выше удержание.

## Что прямо взять в HiDL (без лишнего шума)

1. **Body‑doubling сессии** (Focusmate/Flow/Caveday): «Что делаешь? На сколько?», затем 1 чек‑ин и итог.
2. **Контракт‑режим** (Beeminder/StickK): если не отвечаешь/сливаешь — теряешь доступ к режиму на X часов.
3. **Мини‑ставки и последствия** (Forest/Habitica): не сделал → урона/потери стрика достаточно.
4. **Нулевая нагрузка на пользователя**: минимум текста, максимум «да/нет/перенести».

---

## Паттерны, которые стоит перенять

1. **Body‑doubling как “главный продукт”**: не “ещё функция”, а ядро, куда пользователь заходит ежедневно.
2. **Авто‑планирование вместо вопросов**: предлагать готовые окна времени (2‑3 варианта).
3. **Последствия без унижения**: холодный тон + потеря доступа/привилегий, но без токсичности.
4. **Один экран → одно действие**: “Да/Нет/Перенести” — достаточно.
5. **Пассивные данные > ручной ввод**: меньше «отмечай», больше «я вижу, ты сделал».

## Возможные сценарии для HiDL (MVP)

1. **Кафе фокуса**
   - Выбор задачи и длительности (25/50/90).
   - Один чек‑ин посередине.
   - Итог: сделал / частично / не сделал.
2. **Контракт‑режим**
   - Если не отвечаешь 2–3 раза → тишина на X часов.
   - Срывы снижают “доверие” (менее тёплый тон, меньше бонусов).
3. **Daily‑brief без шума**
   - 1 сообщение в день: 3 ключевых дела + 1 быстрый шаг.
   - Всё остальное — по запросу.

---

## Что избегать

- Слишком много функций в одном меню (теряется фокус).
- Много текстов «как пользоваться» — человек хочет результат без инструкций.
- Уведомления ради уведомлений: лучше 2–4 точечных действия, чем 20 напоминаний.
